\DoxyCodeLine{36 \textcolor{comment}{   *}}
\DoxyCodeLine{37 \textcolor{comment}{   * @section:}}
\DoxyCodeLine{38 \textcolor{comment}{   *   cff\_driver}}
\DoxyCodeLine{39 \textcolor{comment}{   *}}
\DoxyCodeLine{40 \textcolor{comment}{   * @title:}}
\DoxyCodeLine{41 \textcolor{comment}{   *   The CFF driver}}
\DoxyCodeLine{42 \textcolor{comment}{   *}}
\DoxyCodeLine{43 \textcolor{comment}{   * @abstract:}}
\DoxyCodeLine{44 \textcolor{comment}{   *   Controlling the CFF driver module.}}
\DoxyCodeLine{45 \textcolor{comment}{   *}}
\DoxyCodeLine{46 \textcolor{comment}{   * @description:}}
\DoxyCodeLine{47 \textcolor{comment}{   *   While FreeType's CFF driver doesn't expose API functions by itself,}}
\DoxyCodeLine{48 \textcolor{comment}{   *   it is possible to control its behaviour with @FT\_Property\_Set and}}
\DoxyCodeLine{49 \textcolor{comment}{   *   @FT\_Property\_Get.  The list below gives the available properties}}
\DoxyCodeLine{50 \textcolor{comment}{   *   together with the necessary macros and structures.}}
\DoxyCodeLine{51 \textcolor{comment}{   *}}
\DoxyCodeLine{52 \textcolor{comment}{   *   The CFF driver's module name is `cff'.}}
\DoxyCodeLine{53 \textcolor{comment}{   *}}
\DoxyCodeLine{54 \textcolor{comment}{   *   *Hinting* *and* *antialiasing* *principles* *of* *the* *new* *engine*}}
\DoxyCodeLine{55 \textcolor{comment}{   *}}
\DoxyCodeLine{56 \textcolor{comment}{   *   The rasterizer is positioning horizontal features (e.g., ascender}}
\DoxyCodeLine{57 \textcolor{comment}{   *   height \& x-\/height, or crossbars) on the pixel grid and minimizing the}}
\DoxyCodeLine{58 \textcolor{comment}{   *   amount of antialiasing applied to them, while placing vertical}}
\DoxyCodeLine{59 \textcolor{comment}{   *   features (vertical stems) on the pixel grid without hinting, thus}}
\DoxyCodeLine{60 \textcolor{comment}{   *   representing the stem position and weight accurately.  Sometimes the}}
\DoxyCodeLine{61 \textcolor{comment}{   *   vertical stems may be only partially black.  In this context,}}
\DoxyCodeLine{62 \textcolor{comment}{   *   `antialiasing' means that stems are not positioned exactly on pixel}}
\DoxyCodeLine{63 \textcolor{comment}{   *   borders, causing a fuzzy appearance.}}
\DoxyCodeLine{64 \textcolor{comment}{   *}}
\DoxyCodeLine{65 \textcolor{comment}{   *   There are two principles behind this approach.}}
\DoxyCodeLine{66 \textcolor{comment}{   *}}
\DoxyCodeLine{67 \textcolor{comment}{   *   1) No hinting in the horizontal direction: Unlike `superhinted'}}
\DoxyCodeLine{68 \textcolor{comment}{   *   TrueType, which changes glyph widths to accommodate regular}}
\DoxyCodeLine{69 \textcolor{comment}{   *   inter-\/glyph spacing, Adobe's approach is `faithful to the design' in}}
\DoxyCodeLine{70 \textcolor{comment}{   *   representing both the glyph width and the inter-\/glyph spacing}}
\DoxyCodeLine{71 \textcolor{comment}{   *   designed for the font.  This makes the screen display as close as it}}
\DoxyCodeLine{72 \textcolor{comment}{   *   can be to the result one would get with infinite resolution, while}}
\DoxyCodeLine{73 \textcolor{comment}{   *   preserving what is considered the key characteristics of each glyph.}}
\DoxyCodeLine{74 \textcolor{comment}{   *   Note that the distances between unhinted and grid-\/fitted positions at}}
\DoxyCodeLine{75 \textcolor{comment}{   *   small sizes are comparable to kerning values and thus would be}}
\DoxyCodeLine{76 \textcolor{comment}{   *   noticeable (and distracting) while reading if hinting were applied.}}
\DoxyCodeLine{77 \textcolor{comment}{   *}}
\DoxyCodeLine{78 \textcolor{comment}{   *   One of the reasons to not hint horizontally is antialiasing for LCD}}
\DoxyCodeLine{79 \textcolor{comment}{   *   screens: The pixel geometry of modern displays supplies three}}
\DoxyCodeLine{80 \textcolor{comment}{   *   vertical sub-\/pixels as the eye moves horizontally across each visible}}
\DoxyCodeLine{81 \textcolor{comment}{   *   pixel.  On devices where we can be certain this characteristic is}}
\DoxyCodeLine{82 \textcolor{comment}{   *   present a rasterizer can take advantage of the sub-\/pixels to add}}
\DoxyCodeLine{83 \textcolor{comment}{   *   increments of weight.  In Western writing systems this turns out to}}
\DoxyCodeLine{84 \textcolor{comment}{   *   be the more critical direction anyway; the weights and spacing of}}
\DoxyCodeLine{85 \textcolor{comment}{   *   vertical stems (see above) are central to Armenian, Cyrillic, Greek,}}
\DoxyCodeLine{86 \textcolor{comment}{   *   and Latin type designs.  Even when the rasterizer uses greyscale}}
\DoxyCodeLine{87 \textcolor{comment}{   *   antialiasing instead of color (a necessary compromise when one}}
\DoxyCodeLine{88 \textcolor{comment}{   *   doesn't know the screen characteristics), the unhinted vertical}}
\DoxyCodeLine{89 \textcolor{comment}{   *   features preserve the design's weight and spacing much better than}}
\DoxyCodeLine{90 \textcolor{comment}{   *   aliased type would.}}
\DoxyCodeLine{91 \textcolor{comment}{   *}}
\DoxyCodeLine{92 \textcolor{comment}{   *   2) Aligment in the vertical direction: Weights and spacing along the}}
\DoxyCodeLine{93 \textcolor{comment}{   *   y\string~axis are less critical; what is much more important is the visual}}
\DoxyCodeLine{94 \textcolor{comment}{   *   alignment of related features (like cap-\/height and x-\/height).  The}}
\DoxyCodeLine{95 \textcolor{comment}{   *   sense of alignment for these is enhanced by the sharpness of grid-\/fit}}
\DoxyCodeLine{96 \textcolor{comment}{   *   edges, while the cruder vertical resolution (full pixels instead of}}
\DoxyCodeLine{97 \textcolor{comment}{   *   1/3 pixels) is less of a problem.}}
\DoxyCodeLine{98 \textcolor{comment}{   *}}
\DoxyCodeLine{99 \textcolor{comment}{   *   On the technical side, horizontal alignment zones for ascender,}}
\DoxyCodeLine{100 \textcolor{comment}{   *   x-\/height, and other important height values (traditionally called}}
\DoxyCodeLine{101 \textcolor{comment}{   *   `blue zones') as defined in the font are positioned independently,}}
\DoxyCodeLine{102 \textcolor{comment}{   *   each being rounded to the nearest pixel edge, taking care of}}
\DoxyCodeLine{103 \textcolor{comment}{   *   overshoot suppression at small sizes, stem darkening, and scaling.}}
\DoxyCodeLine{104 \textcolor{comment}{   *}}
\DoxyCodeLine{105 \textcolor{comment}{   *   Hstems (this is, hint values defined in the font to help align}}
\DoxyCodeLine{106 \textcolor{comment}{   *   horizontal features) that fall within a blue zone are said to be}}
\DoxyCodeLine{107 \textcolor{comment}{   *   `captured' and are aligned to that zone.  Uncaptured stems are moved}}
\DoxyCodeLine{108 \textcolor{comment}{   *   in one of four ways, top edge up or down, bottom edge up or down.}}
\DoxyCodeLine{109 \textcolor{comment}{   *   Unless there are conflicting hstems, the smallest movement is taken}}
\DoxyCodeLine{110 \textcolor{comment}{   *   to minimize distortion.}}
\DoxyCodeLine{111 \textcolor{comment}{   *}}
\DoxyCodeLine{112 \textcolor{comment}{   * @order:}}
\DoxyCodeLine{113 \textcolor{comment}{   *   hinting-\/engine}}
\DoxyCodeLine{114 \textcolor{comment}{   *   no-\/stem-\/darkening}}
\DoxyCodeLine{115 \textcolor{comment}{   *   darkening-\/parameters}}
\DoxyCodeLine{116 \textcolor{comment}{   *}}
\DoxyCodeLine{117 \textcolor{comment}{   */}}
\DoxyCodeLine{118 }
